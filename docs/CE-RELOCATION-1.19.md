# HOI4 1.19 signature-regeneration runbook (Cheat Engine, by hand)

> Cheat Engine cannot run headless — these are the exact steps the maintainer runs
> against the live `hoi4.exe`. Generated by the `signature-finder` pass; companion
> to the general `docs/METHODOLOGY.md`.

The three scripts you drive:
- `lua/healthcheck.lua`
- `lua/find_player_base.lua`
- `lua/aobgen.lua`

They share `lua/lib/aoblib.lua` and read the baseline catalog `data/signatures.lua`.

---

## 1. Preconditions

1. Launch **Hearts of Iron IV 1.19** and **load a save** (a live campaign, not the
   main menu) — the scripts need every system resident in memory; `healthcheck.lua`
   itself warns to "load a save so all systems are in memory". Work on a *copy* of
   the save (bad writes corrupt campaigns).
2. Open **Cheat Engine 7.4+** (scripts target CE 7.x; `aoblib.lua` uses
   `enumModules`/`process`, `createMemScan`, etc.).
3. Attach CE: click the flashing computer icon (top-left) → pick **`hoi4.exe`** →
   **Open**. `healthcheck.lua` aborts with `[healthcheck] No process attached` if
   `getOpenedProcessID() == 0`.
4. Open the **Lua Engine console** (where every command below is pasted):
   - Main CE window: **Table → Lua Engine…**, or
   - Memory View (Ctrl+Alt+M): **Tools → Lua Engine** (Ctrl+Alt+L).
   Type/paste in the lower input box, press **Execute**.
5. Set the toolkit path once per session:
   ```lua
   TOOLKIT_DIR = [[C:\Users\Brock\Documents\My Cheat Tables\HOI4-Memory-Toolkit]]
   ```

The GUI steps in §3 ("Find out what accesses this address") happen in the **Memory
View** window — keep both windows open.

## 2. Health check — what actually broke

```lua
TOOLKIT_DIR = [[C:\Users\Brock\Documents\My Cheat Tables\HOI4-Memory-Toolkit]]
dofile(TOOLKIT_DIR..[[\lua\healthcheck.lua]])
```

Loads the 26-entry catalog from `data/signatures.lua` and runs
`aob.scanInModule(s.pattern)` for each (an `AOBScan` filtered to the `hoi4.exe`
module range). One line per symbol, then a summary.

Reading the output:
- `[ ok ]` — pattern resolved to **exactly 1** hit. Healthy; address shown via
  `aob.fmt`, e.g. `hoi4.exe+1A2B3C4 (0x...)`.
- `[warn] AMBIGUOUS xN` — matched **N>1** times; no longer unique → re-mint with
  `aobgen.lua` (§3c).
- `[FAIL] MISSING` — **0** hits; code moved/changed → relocate from scratch.
- A trailing `*ANCHOR*` marks the three anchor scans: **MOHP, MOSF, MOSR**.

Record the summary: `Resolved <ok>/26   Missing <m>   Ambiguous <a>`, plus the
BROKEN and AMBIGUOUS lists.

What "broken" means per anchor:
- **MOHP broken** → the **player struct (`pPlayer`)** is unresolved. Breaks the
  whole PP / Stability / War Support / Command Power / XP family (all offsets off
  `pPlayer`). **Fix this first.**
- **MOSF broken** → the selected **Army/Fleet** anchor (`pArmyFleet`) is gone.
- **MOSR broken** → the selected **Province/region** anchor (`pRegion`) is gone.

Relocate the `*ANCHOR*` scans first; dependent cheats re-point off them via the
offsets in `data/signatures.lua → M.knownOffsets`.

## 3. Anchor relocation (MOHP, then MOSF / MOSR)

### 3a. Re-find the player base (MOHP / `pPlayer`) via guided value scan

```lua
TOOLKIT_DIR = [[C:\Users\Brock\Documents\My Cheat Tables\HOI4-Memory-Toolkit]]
dofile(TOOLKIT_DIR..[[\lua\find_player_base.lua]])   -- defines pbFirst, pbNext, pbList
```

Political Power is stored **×1000** (4-byte int). Read your current PP off the HUD
(e.g. 700 PP → scan `700000`):
```lua
pbFirst(700000)     -- firstScan(soExactValue, vtDword, rtRounded, "700000", ...)
```
Change PP in-game (spend on a decision/advisor, or wait for daily gain), read the
new value, narrow:
```lua
pbNext(685000)      -- nextScan(soExactValue, vtDword, rtRounded, "685000", ...)
```
Repeat `pbNext(<current PP ×1000>)`. When the found-list drops to **≤ 8**, `pbNext`
auto-calls `pbList()`; otherwise call it:
```lua
pbList()            -- prints up to 20 surviving addresses with indices
```
Confirm: change PP once more; exactly one candidate should track it. That address
holds PP (= `[[pPlayer+0xEA0]+0xC8]`).

### 3b. Recover `pPlayer` from the confirmed address (Memory View GUI)

1. Right-click the surviving address → **Find out what accesses this address**.
2. Trigger it in-game (open politics / let PP tick).
3. You'll see e.g. `mov [rax+000000C8],ecx`. Since PP = `[[pPlayer+0xEA0]+0xC8]`,
   the base register (`rax`) = `[pPlayer+0xEA0]`.
4. Walk back **one indirection** to `pPlayer`: "Find out what accesses" / pointer
   scan until you reach the pointer whose value `+0xEA0` lands on `rax`. That is
   `pPlayer`.
5. Cross-check by reading sibling offsets off the recovered `pPlayer` (should read
   sane ×1000 values): Stability `[pPlayer+0xFA8]`, War Support `[pPlayer+0xFAC]`,
   Command Power `[pPlayer+0x1C4]`.

### 3c. Mint a fresh, patch-tolerant AOB for the injection point

Pick the instruction address to hook (from the "what accesses/writes" window — the
original MOHP hooks `41 8B 96 C4 01 00 00 …`, `mov edx,[r14+000001C4]`; choose the
1.19 equivalent):
```lua
TOOLKIT_DIR = [[C:\Users\Brock\Documents\My Cheat Tables\HOI4-Memory-Toolkit]]
AOBGEN_ADDR = "hoi4.exe+1A2B3C4"   -- real address from Memory View
AOBGEN_LEN  = 24                    -- optional; default 24 bytes
dofile(TOOLKIT_DIR..[[\lua\aobgen.lua]])
```
`aobgen.lua` resolves with `getAddress`, reads via `readBytes(addr, L, true)`,
**wildcards the rel32 operand** of any `E8`/`E9`/`0F 8x` (those displacements drift),
and grows the pattern from `AOBGEN_LEN` up to `+16` bytes until `aob.scanInModule`
returns exactly one hit. Look for:
```
signature : <hex with ?? masks>
status    : UNIQUE in hoi4.exe -- safe to use in AOBScanModule().
```
If `status : N matches -- NOT unique yet`, raise `AOBGEN_LEN` (32, 40) or pick a more
distinctive instruction. Copy the `signature` line — that's the new **MOHP** pattern.

### 3d. Repeat for MOSF and MOSR

Each is its own code hook (not a `pPlayer` offset):
- **MOSF (`pArmyFleet`)**: select an army/fleet; "Find out what accesses" on its
  selected-unit field; land on the instruction matching old `48 8B 82 E0 01 00 00 …`;
  run `aobgen.lua` with `AOBGEN_ADDR` set to it.
- **MOSR (`pRegion`)**: select a province/region; same loop for the instruction
  matching old `49 8B 86 20 01 00 00 …`.

After each, re-run `healthcheck.lua` and confirm the symbol flips to `[ ok ]`.

## 4. Re-anchor the high-value cheats

### A. Offsets off `pPlayer` (no new AOB once MOHP is fixed)

Once §3a–3c give a reliable `pPlayer`, these reuse it via
`data/signatures.lua → M.knownOffsets`:

| Cheat | Offset | Drift risk in 1.19 |
|---|---|---|
| **Political Power** | `[[pPlayer+0xEA0]+0xC8]` (int, ×1000) | inner `+0xC8` stable; **outer `+0xEA0` pointer most likely to drift** — re-verify (§3b) |
| Stability | `[pPlayer+0xFA8]` (int, ×1000) | large offset, moderate drift — verify with War Support |
| War Support | `[pPlayer+0xFAC]` (int, ×1000) | adjacent to Stability; verify together |
| Command Power | `[pPlayer+0x1C4]` (int, ×1000) | small offset, usually stable |
| Army XP | `[pPlayer+0x1E0]` (int, ×32768) | small offset, usually stable |
| Navy XP | `[pPlayer+0x1F8]` (int, ×32768) | small offset, usually stable |
| Air XP | `[pPlayer+0x210]` (int, ×32768) | small offset, usually stable |
| (player id filter) | `[pPlayer+0x18] & 0x00FFFFFF` | usually stable |

Rule of thumb (`docs/METHODOLOGY.md`): struct field offsets survive minor patches;
small offsets near the struct head (`0x18`, `0x1C4`, `0x1E0/0x1F8/0x210`) are safest.
Re-confirm the large offsets (`0xEA0`, `0xFA8`, `0xFAC`) by reading them back off the
recovered `pPlayer` before trusting them.

### B. Cheats that are their own code injection (each needs an `aobgen.lua` pass)

If `healthcheck.lua` shows these `[FAIL]`/`[warn]`, relocate the individual
instruction ("Find out what writes/accesses" on the in-game value) and mint a fresh
pattern as in §3c:
- **Research** → `MORP` ("Research in 1 day") — research-progress write.
- **National Focus** → `MOFP` ("Finish focus in 1 day").
- **Manpower** → `MOMM` ("Minimum manpower per state", 7000) — longest baseline
  (39 bytes, 9 wildcards); set `AOBGEN_LEN` higher (~40) when re-minting.
- **God Mode** → `GDMD` (army HP+org), `GMDS` (ships strength+manpower), `GDS2`
  (ships org). Trigger combat so the code path executes before "Find out what
  accesses".

Code signatures break easily (any nearby instruction change shifts bytes) — relocate
the *pointer* once and lean on stored offsets where possible.

## 5. Capturing the new 1.19 signatures back into the toolkit

`data/signatures.lua` / `.json` are **auto-generated — do NOT hand-edit** (header +
CLAUDE.md enforce it). Source of truth is the feature/offset map in
`tools/extract_baseline.py` (it parses `AOBScanModule(symbol,$process,pattern)` lines
out of a Recifense-style `.CT`).

1. Paste each newly-minted `UNIQUE` signature into your working `.CT` as
   `AOBScanModule(SYMBOL,$process,<pattern>)` (MOHP, MOSF, MOSR, MORP, MOFP, MOMM,
   GDMD, GMDS, GDS2, …).
2. If an offset changed during verification, update `KNOWN_OFFSETS` in
   `extract_baseline.py`; if you added/renamed a symbol, update its `FEATURES` map.
   Anchors are hard-coded as `is_anchor = sym in ("MOHP","MOSF","MOSR")`.
3. Regenerate:
   ```bash
   python tools/extract_baseline.py "path/to/Hearts of Iron IV.CT" --game-version 1.19.x
   ```
   Rewrites both `data/signatures.lua` and `.json` (bumps `gameVersion`/`generated`).
4. Re-run `healthcheck.lua`: every relocated symbol should read `[ ok ]`, summary
   `Resolved 26/26  Missing 0  Ambiguous 0`.
5. Keep the old 1.11.10 catalog in git history so a diff shows exactly which
   patterns/offsets 1.19 moved.
